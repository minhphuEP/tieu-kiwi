Tieu Kiwi — Architecture

An AI Quality-Engineering agent in Slack. It answers QE questions by reasoning over two knowledge stores — a Postgres knowledge graph (structural truth) and a Chroma vector store (semantic recall via RAG) — on the Anthropic Claude API (claude-sonnet-4-6). This page documents what the code actually does.

Layer A — Agent core Layer B — Slack wrapper Layer C — Feedback loop Postgres graph Chroma RAG Jira / Confluence

2Graph data model (ontology)

Stored relationally in nodes(id, type, ref, project_id, props_json) + edges(id, src_id, rel, dst_id), queried with parameterized SQL — no Neo4j. project_id = Jira key prefix (CDM-268 → CDM).

An AcceptanceCriterion with no coveredBy edge = a coverage gap. go_no_go combines coverage gaps + failing TestRuns + open critical/high Bugs. Edges may cross projects (e.g. Component dependsOn Component); nodes carry project_id, edges don't.

⚠ Real-behaviour note: fetch_jira upserts the Requirement node without setting project_id (it lands NULL). The project_id_from_ref helper (Jira-prefix) is applied when save_testcases writes TestCase nodes, and the deterministic Slack shortcuts pass the channel's project explicitly — so scoping still works for those flows.

📥 Fetch ticket

@Tieu Kiwi fetch CDM-268

The agent loop calls fetch_jira, reads the Jira issue (REST v3) and upserts a Requirement node with metadata: status, issuetype, priority, assignee/reporter, story points, description. ACs are seeded separately — they are not parsed from Jira.

Store / tool Jira REST → Graph fetch_jira → Requirement node saved with ticket metadata

🚦 Go / No-Go

@Tieu Kiwi CDM-268 đã đủ điều kiện go live chưa?

Deterministic go_no_go over the graph. GO only if no coverage gaps and no failing tests and no open critical/high bug. Reports coverage, failing tests, open bugs + next_actions.

Store / tool Postgres graph go_no_go On GO → @mentions Delivery Manager + Approve / Reject sign-off → GO / NO-GO decision recorded in go_live_decisions

🧪 Coverage / gen testcase

@Tieu Kiwi generate testcase cho CDM-268 @Tieu Kiwi write testcase for CDM-268

Drafts test cases covering every AC of the requirement (not a single AC), exports an Excel file (Testomat.io format), posts a draft + Approve / Refine buttons. Approve → saves TestCase nodes + coveredBy edges; Refine → LLM re-draft (v+1).

Store / tool Graph write + RAG gen_testcase · testcase_gen @mentions QE Lead as approver → Excel draft → approved TestCase nodes with _meta provenance

❓ Clarify / PO-confirm

@Tieu Kiwi CDM-268 cần PO chốt: ...

find_ambiguities reads the requirement (fetching Jira + linked Confluence PRD if needed) and flags genuine ambiguities against 3 dimensions (Behaviour & Edge Cases, Constraints, Conflicts), phrased as direct PO questions, with an "Open clarification form." Submitted answers are written back onto the Requirement node.

Store / tool RAG rubric + LLM find_ambiguities @mentions PO → answers stored as clarified_requirements on the requirement

🎓 Curator + learn rule

@Tieu Kiwi curator-test @Tieu Kiwi học rule mới: <rule>

Enqueues a candidate rule in promotion_queue and shows Approve / Edit / Reject. Approve = promotion: the rule is inserted into kb_rules (status active) and indexed into Chroma, so search_kb can retrieve it thereafter.

Store / tool Queue → Chroma add_candidate_rule · approve_candidate @mentions QE Lead as curator → new team rule, retrievable via RAG

The agent answers the bug / failing-test question from the graph, then the Slack layer appends a @mention of the Dev owner. (The classify_bug and bug_blast_radius tools are also available for deeper triage in free-form Q&A.)

5Two stores, two question types

The core design choice: use the store that fits the question.

Postgres graph — structural truth

Exact, joinable facts and traceability. An AC either has a covering TestCase or it doesn't; a TestRun passed or failed; a bug blocks release or it doesn't. Best when the answer is a relationship or an aggregate over known entities.

ANSWERS QUESTIONS LIKE
  • Is CDM-268 ready to ship? (go_no_go)
  • Which ACs have no tests? (coverage_gap)
  • Trace this requirement end-to-end. (trace)
  • How big is this bug's blast radius? (bug_blast_radius)

Chroma RAG — semantic recall

Fuzzy retrieval over prose where no exact key exists: team rules, PRD/BRD text, glossaries, test-case templates & rubrics. Best when the answer is "what does the doc / rule say?" Grows over time via curator promotion. Isolated behind rag.py (local all-MiniLM-L6-v2 embedding).

ANSWERS QUESTIONS LIKE
  • What's our test-case naming rule? (search_kb)
  • What does the PRD say about rollout gating?
  • Which rubric applies to writing this test case? (inside gen_testcase)
